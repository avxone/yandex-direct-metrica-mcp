# QA: v2.0.1 MCP tool smoke test (Claude Code)

Date: 2026-02-05

Scope: verify tool stability and identify MCP bugs/regressions in the currently published `yandex-direct-metrica-mcp-pro` release. Read-only checks only.

Test profile:
- `account_id`: `elama-16161182_vx` (Headset VX)
- `direct_client_login`: `elama-16161182`
- `metrica_counter_id`: `91450749`
- Date window: `2026-01-21` → `2026-02-03` (and W05 slice `2026-01-26` → `2026-02-01`)

## Completed

- Verified basic connectivity:
  - `accounts.list`: OK, account found.
  - `metrica.list_counters(account_id=...)`: OK, counter `91450749` present.
  - `wordstat.user_info`: OK.
  - `audience.segments.list(limit=1)`: OK (empty list, API responds).

- Found a Direct tool regression:
  - `direct.list_clients(account_id="elama-16161182_vx")` → **ERROR**
    - Error type: `YandexDirectClientError` (code `8000`)
    - Message: `Invalid request: Unknown parameter specified: SelectionCriteria`
    - `request_id`: `6157324516967202494`
    - Hypothesis: the `clients` endpoint does not accept `SelectionCriteria`, but our helper `_build_basic_params()` always sends it (even `{}`).

- Observed schema/UX issues:
  - `direct_client_login` schema appears as `enum=["georgiyagaev"]` even though account registry uses other logins (`elama-*`, `e-*`).
  - Impact: LLM/tooling may refuse valid `direct_client_login` overrides in agency scenarios.
  - Note: runtime workaround is to use `account_id` resolution, but schema should not block explicit overrides.

- Observed expected-but-confusing behavior in pressure report:
  - `direct.hf.pressure_report(..., placement="search", include_breakdown=true)` returns `status="partial"` with fallback warning:
    - `Direct report fallback: SEARCH_QUERY_PERFORMANCE_REPORT does not support requested filters/dimensions (YandexDirectClientError).`
    - `by_cluster_region_device` is `null` (breakdown not available after fallback).
  - `direct.hf.pressure_report(..., placement="all", include_breakdown=false)` returns `status="ok"`, totals identical.
  - Conclusion: breakdown/dimensions are best-effort; we should document limitations and/or improve fallback semantics.

- Hit large-response UX limitation:
  - `direct.hf.report_search_phrases(date_from="2026-01-26", date_to="2026-02-01")` produced ~`317 KB` payload; Claude Code saved it to a file due to token limits.
  - This is not a functional bug, but harms usability for interactive analysis.

## To Do

### P1: Fix `direct.list_clients` request shape

Options:
1) **Special-case** `direct.list_clients`: build params without `SelectionCriteria` (safest, minimal blast radius).
2) Make `_build_basic_params()` **omit `SelectionCriteria` when empty** (broader change; likely OK but touches multiple endpoints).
3) Add a new helper `_build_basic_params_no_selection()` and use it for endpoints that reject the field (explicit and safe).

Acceptance criteria:
- `direct.list_clients` works against agency accounts and personal accounts.
- No regressions in `direct.list_campaigns`, `direct.list_adgroups`, `direct.list_ads`, etc.

### P1: Relax `direct_client_login` schema (do not hard-enforce enum)

Options:
1) Change schema to `anyOf: [{enum:[known...]},{type:"string"}]` for hinting without restricting input.
2) Remove `enum` entirely; keep only `type:"string"` and a description listing known logins.
3) Populate the enum from **account registry** (`accounts.json`) instead of env vars (better defaults for multi-profile setups).

Acceptance criteria:
- Tools remain callable with any valid `direct_client_login`, while still suggesting known values.

### P2: Improve HF report UX for large outputs

Options:
1) Add optional parameters to `direct.hf.report_search_phrases`: `limit`, `offset`, `top_by` (cost/impressions/clicks).
2) Add `aggregate=true` mode: group by `Query` and return compact rows (CTR/CPC).
3) Document a “use datasets” path and steer users to `dashboard.dataset.direct_search_phrases_daily` for BI/exports.

Acceptance criteria:
- Interactive use stays under typical tool output limits for common date windows.

### P2: `direct.report` defaults (UX)

Observed in public smoke: `direct.report` requires several parameters to be passed explicitly (`date_range_type`, `format`, `include_vat`) even when safe defaults exist.

Proposed change:
- Default in the server-side builder when omitted:
  - `DateRangeType="CUSTOM_DATE"`
  - `Format="TSV"`
  - `IncludeVAT="YES"`
  - `IncludeDiscount="NO"`

Acceptance criteria:
- Minimal call with `report_type`, `field_names`, `date_from`, `date_to` succeeds without extra params.

### P2: Audience lookalikes tools (404)

Observed: `audience.lookalikes.list` returns HTTP 404 in public smoke (endpoint likely unsupported / incorrect).

Options:
1) Remove `audience.lookalikes.*` tools from the surface (recommended if API truly lacks endpoints).
2) Keep tools but return a structured `not_supported` error with a hint to use `audience.segments.list` (if lookalikes are represented as segments).
3) Implement a fallback to segments listing + filtering when possible.

Acceptance criteria:
- No persistent 404 errors from exposed tools in default public usage.
