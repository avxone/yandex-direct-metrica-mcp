Плейбук запуска diagnose → plan → review → apply
Все демо Лендинг

Запуск кампании без “black‑box”

Это пример алгоритма, который удобно демонстрировать клиентам: вы формируете гипотезы, строите план изменений, проходите ревью, применяете под guardrails и измеряете эффект. Инструменты — ваши (LLM/локальные модели), данные и артефакты — у вас.

Public — диагностика и отчёты PRO — plan/apply и BI sync Артефакты — доказуемость результата

0) Контекст (что спросить у клиента)

Минимум

  • ACCOUNT_ID (или direct_client_login + counter_id)
  • Период: DATE_FROM → DATE_TO (обычно “до вчера”)
  • Главные KPI: заявки/продажи/визиты, и какие цели считать (goal_ids)
  • Ограничения: бюджеты/сроки/гео/устройства

Флаги безопасности (PRO writes)

MCP_PUBLIC_READONLY=false
MCP_WRITE_ENABLED=true
HF_WRITE_ENABLED=true
MCP_WRITE_SANDBOX_ONLY=true
YANDEX_DIRECT_SANDBOX=true

# optional:
MCP_TWO_PHASE_WRITES=true

По умолчанию рекомендуем sandbox‑only и двухфазное подтверждение.

1) Diagnose (read‑only): быстро понять, что происходит

Дашборд (для человека)

{
  "tool":"dashboard.generate_option1",
  "arguments":{
    "account_id":"${ACCOUNT_ID}",
    "date_from":"${DATE_FROM}",
    "date_to":"${DATE_TO}",
    "output_dir":"/data/dashboards",
    "return_data":false,
    "include_raw_reports":false
  }
}

Проблемные зоны (быстро)

{
  "tool":"direct.hf.get_campaign_summary",
  "arguments":{
    "account_id":"${ACCOUNT_ID}",
    "limit":50
  }
}

Дальше обычно: search phrases → Wordstat → минуса.

2) Plan (PRO, preview): собрать план изменений

Пример: добавить ключи + слить минуса (без применения)

{
  "tool":"direct.hf.plan_changes",
  "arguments":{
    "account_id":"${ACCOUNT_ID}",
    "operations":[
      {
        "op":"keywords.add",
        "adgroup_id":123456,
        "keywords":[
          {"keyword":"купить газонокосилку"},
          {"keyword":"газонокосилка бензиновая"}
        ]
      },
      {
        "op":"negatives.merge",
        "campaign_id":987654,
        "negatives":["б/у","ремонт","инструкция","чертеж"]
      }
    ]
  }
}

Ожидайте в ответе preview + warnings. Это удобно показывать клиенту как “план работ”.

3) Review: ревью человеком (обязательный этап)

Чеклист ревью

  • Правильный аккаунт/кампания/группа
  • Согласованы KPI и цели (goal_ids)
  • Минуса не убьют бренд/коммерческий интент
  • Ограничения по бюджету и срокам

Артефакты

  • Сохраняйте ответы tool calls (JSON) в папку запуска
  • Фиксируйте гипотезы, изменения и “почему так”
  • Измеряйте эффект в согласованном окне (до/после)

4) Apply (PRO): применить изменения под guardrails

Применение (explicit apply=true)

{
  "tool":"direct.hf.apply_plan",
  "arguments":{
    "account_id":"${ACCOUNT_ID}",
    "apply":true,
    "plan_id":"${PLAN_ID_FROM_PREVIEW}"
  }
}

Если включён two‑phase режим, часть write‑операций может потребовать отдельного подтверждения через write.confirm.

5) Measure: измерить эффект и закрыть гипотезу

Снимок после (дашборд)

{
  "tool":"dashboard.generate_option1",
  "arguments":{
    "account_id":"${ACCOUNT_ID}",
    "date_from":"${AFTER_FROM}",
    "date_to":"${AFTER_TO}",
    "output_dir":"/data/dashboards",
    "return_data":false,
    "include_raw_reports":false
  }
}

BI (если нужно)

{
  "tool":"dashboard.sync.start",
  "arguments":{
    "datasets":[
      "dashboard.dataset.direct_campaign_daily",
      "dashboard.dataset.metrica_daily"
    ],
    "account_ids":["${ACCOUNT_ID}"],
    "date_from":"${DATE_FROM}",
    "date_to":"${DATE_TO}",
    "page_size":500,
    "chunk_days":7
  }
}
